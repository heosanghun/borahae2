# K-뷰티 얼굴 연계 기획: 전문가 분석 및 개선안

> 기존 전략·Phase 1 설계 문서를 분석한 뒤, **문제점·보완점·제안점**을 나열하고, 개선 방향을 정리한 문서입니다.

---

## 1. 문제점 (Problems)

### 1-1. 핵심 플로우 불일치: "얼굴 업로드" vs 실제 입력

| 문제 | 설명 |
|------|------|
| **얼굴 → 톤 자동 분석 부재** | 전략은 "얼굴 업로드 → 얼굴에 맞는 제품"인데, 현재는 **Step 4에서 사용자가 직접** 피부톤·언더톤을 선택함. 얼굴 사진으로 AI가 톤을 추정하는 단계가 없음. |
| **결과** | "얼굴에 맞는"이 사실상 **사용자 자기 선언**에 의존하며, 외국인·비전문가에게는 선택이 어렵고 오류 가능성이 큼. |

**개선 방향**:  
- 단기: "우리 분석은 선택하신 톤을 기준으로 합니다"라고 명시하고, Step 2에서 "사진을 올리시면 추후 자동 톤 분석으로 개선될 예정" 안내.  
- 중기: 얼굴 이미지 → 톤 추정 API 또는 내부 모델 도입 로드맵 수립.

---

### 1-2. 데이터 소스 이중화: Step 4 입력 vs AI 분석 결과

| 문제 | 설명 |
|------|------|
| **두 가지 톤 소스** | Step 4: `stylingData.skinTone`, `stylingData.undertone` (사용자 선택). Step 5: `result.personalColor.season` (AI 생성). |
| **불일치 가능성** | AI가 Step 4 입력을 반영해 season을 만들지, 별도 프롬프트만 쓰는지 문서에 없음. 사용자는 "여름쿨"인데 AI가 "가을웜"을 줄 수 있음. |
| **매칭 혼란** | K-뷰티 매칭에 "season만 쓸지", "undertone도 쓸지" 명확하지 않으면 같은 사용자에게 서로 다른 추천이 나올 수 있음. |

**개선 방향**:  
- **매칭 입력 정의 고정**: "Phase 1에서는 **AI 분석 결과 `result.personalColor.season`만** 사용하고, `stylingData.undertone`은 보조(동점 시 우선순위)로만 사용" 등 한 가지 규칙을 정해 설계·코드에 반영.  
- AI 분석 시 Step 4 입력을 반영하도록 프롬프트/로직 명시.

---

### 1-3. 개인정보·동의·광고 표시

| 문제 | 설명 |
|------|------|
| **동의 구체화 부족** | "맞춤 제품·광고를 위해 분석 결과를 활용합니다" 수준만 있고, **어느 단계에서** 어떤 문구로 동의를 받을지, 체크박스/버튼 위치가 없음. |
| **법적 요건 미반영** | 한국 개인정보보호법, 전자상거래법(광고 표시), EU GDPR(동의·목적 제한) 등에 맞는 문구·절차가 없음. |
| **광고/제휴 라벨** | "추천 vs 광고 구분"만 있고, 한국·해외에서 요구하는 **"광고"·"제휴"** 표시 문구 예시가 없음. |

**개선 방향**:  
- 분석 시작 전 또는 Step 4 진입 시 "맞춤 K-뷰티 추천 및 제휴 링크 안내를 위해 톤·스타일 결과를 활용합니다" 문구 + 동의 체크.  
- 제품 카드 근처에 "제휴 링크 포함" 또는 "Ad" 등 짧은 라벨 추가.  
- 별도 문서에 "개인정보·광고 표시 가이드" 초안 작성(아래 보완점 반영).

---

### 1-4. 제품 데이터·운영 리스크

| 문제 | 설명 |
|------|------|
| **제품 정보 신뢰성** | 하드코딩/수동 등록 시 이미지 URL 만료, 상품 페이지 이동·삭제 시 깨진 링크 발생. |
| **관리 주체 불명** | 제품 목록을 누가, 어떤 주기로 갱신할지 정의되지 않음. |
| **카테고리 편중** | 립·블러셔 위주이고, K-뷰티 비중이 큰 **스킨케어**(세럼, 크림, 선크림)와 **피부 타입**(지성/건성/민감)이 없음. |

**개선 방향**:  
- 제품 DB 관리 주체·업데이트 주기(분기 1회 등) 정하고, 링크 유효성 검사(정기 체크) 계획 수립.  
- Phase 2에서 스킨케어·피부 타입 태그 추가.

---

### 1-5. 매칭 실패·엣지 케이스

| 문제 | 설명 |
|------|------|
| **매칭 0건** | 해당 season+undertone 조합에 제품이 없을 때 빈 카드 또는 에러 노출 가능. |
| **fallback 미정의** | "아무 제품 2개" fallback 시 사용자 톤과 무관한 제품이 나오면 신뢰 하락. |

**개선 방향**:  
- 0건이면 카드 내 문구만 표시: "이 톤에 맞는 제품을 준비 중이에요. 곧 더 많은 제품을 만나보세요." + (선택) 일반 인기 제품 1개 또는 CTA.  
- fallback은 "neutral 톤 제품"만 노출하거나, "인기 K-뷰티" 고정 1개로 제한.

## 3. 제안점 (Suggestions)

### 3-1. 검증·단계 완화

| 제안 | 설명 |
|------|------|
| **Phase 0 옵션** | Phase 1 전에 "K-뷰티 추천" 카드만 노출하고, 실제 매칭 대신 "맞는 제품을 준비 중이에요" + 이메일 수집 또는 "알림 신청" CTA로 관심도·트래픽 검증. |
| **A/B 테스트** | K-뷰티 카드 유무에 따른 이탈률·클릭률·전환률 측정 후, 본격 제품 연동 여부 결정. |

### 3-2. 파트너십·수익화

| 제안 | 설명 |
|------|------|
| **1~2개 브랜드 선계약** | 실제 제품 데이터·제휴 링크를 받은 뒤 Phase 1을 실추천으로 오픈하면, 링크 품질·수익화 검증이 쉬움. |
| **리타겟팅** | (동의 시) 분석 결과 세그먼트(예: 여름쿨)를 익명화해 페이스북/구글 등 광고 플랫폼 리타겟팅에 활용. |

### 3-3. 콘텐츠·외부 유입

| 제안 | 설명 |
|------|------|
| **UGC 연계** | "이 톤으로 찍은 셀카" 공유 유도 후 K-뷰티 해시태그·브랜드와 연계. |
| **미니 가이드** | "K-beauty for your skin tone" 짧은 블로그/영상 + 추천 제품 리스트로 검색·SNS 유입. |

### 3-4. 기술·데이터

| 제안 | 설명 |
|------|------|
| **제품 API** | 장기적으로 제품 목록을 CMS 또는 파트너 API로 관리하면, 링크·이미지 갱신이 수월. |
| **클릭·노출 로깅** | 제품별 노출/클릭 수 집계(익명)해 어떤 톤·제품이 전환에 유리한지 분석. |
